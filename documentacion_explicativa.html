[ REPORTE_DE_ERROR ] Contexto del Proyecto

Problema

La empresa gestiona múltiples clientes y campañas de marketing (SEO, redes sociales, anuncios), pero actualmente lleva el seguimiento en Excel. Esto provoca:

  • Dificultad para saber qué campañas están activas.
  • Falta de control sobre tareas asignadas a empleados.
  • Problemas para ver resultados (KPIs) por campaña.
  • Duplicidad de información.

>>> Ejemplo en la vida real:

"Un cliente contrata una campaña de SEO y al mismo tiempo otra de Anuncios Pagados. El equipo encargado de los anuncios (Empleado A) anota los reportes en su propia hoja de cálculo, mientras que el equipo de SEO (Empleado B) los guarda en un documento distinto. A fin de mes, el coordinador pierde horas intentando cruzar la información para generar el reporte de KPIs total para el cliente. Peor aún, como toda la información está dispersa, frecuentemente se duplican los datos del cliente, la campaña queda listada indefinidamente como 'activa' y no se sabe exactamente qué tarea de optimización web le tocaba hoy al Empleado B."

[ MODELO_DATOS ] Arquitectura Relacional

Diagrama Entidad-Relación (ER)

El siguiente modelo relacional ilustra la estructuración en base de datos de las entidades y cómo se vinculan. Está renderizado nativamente a través del código fuente con MermaidJS:

erDiagram CLIENTE { int id_cliente PK string nombre string email string telefono string empresa } CAMPAÑA { int id_campania PK string nombre string tipo date fecha_inicio date fecha_fin int id_cliente FK } EMPLEADO { int id_empleado PK string nombre string email string rol } TAREA { int id_tarea PK string descripcion string estado date fecha_limite int id_campania FK int id_empleado FK } RESULTADO { int id_resultado PK float valor date fecha int id_campania FK } CLIENTE ||--o{ CAMPAÑA : tiene CAMPAÑA ||--o{ TAREA : incluye EMPLEADO ||--o{ TAREA : realiza CAMPAÑA ||--o{ RESULTADO : genera

>>> Análisis de Relaciones e Independencia:

  • CLIENTE ↔ CAMPAÑA (1:N): Un CLIENTE único puede poseer múltiples instancias de CAMPAÑA asociadas, centralizando sus proyectos en un registro unificado (e.g., eliminando la duplicidad comentada en el ejemplo).
  • CAMPAÑA ↔ RESULTADO (1:N): Una CAMPAÑA genera constantemente puntos de RESULTADO (KPIs y métricas). Cada métrica queda iterativamente anclada a su respectiva campaña matriz mediante id_campania, aportando trazabilidad temporal exacta y facilitando la elaboración de reportes mensuales.
  • TAREA como Entidad Pivote: La tabla TAREA vincula las acciones a ejecutar (id_campania) con quién debe responsabilizarse de hacerlas (id_empleado). De esta manera un empleado realiza múltiples tareas, y una campaña aglutina diversas asignaciones, obteniendo completo control de estatus sobre la fase de cada equipo operativo.

[ SYS_DEPLOY ] Módulos Implementados

A través del backend en Spring Boot, hemos encapsulado la lógica del negocio en dominios relacionales, centralizando todos los subprocesos lógicos y la trazabilidad descrita en el esquema a través de APIs de red:

[ MOD_01 ]

Apartado de Clientes

¿Qué soluciona?: La duplicidad de datos.

Implementación: Entidad central conectada a la BD dedicada a la identidad corporativa. Mantiene el directorio estructurado y limpio de redundancias semánticas.

[ MOD_02 ]

Apartado de Campañas

¿Qué soluciona?: Dificultad de rastreo de estatus.

Implementación: Funciona como un pipeline aglutinador. Posee fechas estrictas e indicadores lógicos de viabilidad (activa/inactiva/finalizada).

[ MOD_03 ]

Apartado de Empleados

¿Qué soluciona?: Desorganización del recurso humano.

Implementación: Instancia al directorio de perfiles y roles que ejecutan trabajo en la agencia, proveyendo autoría trazable sobre quién ejecuta el trabajo.

[ MOD_04 ]

Apartado de Tareas

¿Qué soluciona?: Falta de control en la delegación de responsabilidades.

Implementación: Vincula en forma de puente automatizado a un Empleado con una Campaña específica, imponiendo control de avance semafórico y prioridades asíncronas.

[ MOD_05 ]

Resultados / Métricas

¿Qué soluciona?: Problemas para obtener KPIs y compilar estadísticos unificados.

Implementación: Recolector en forma de logs y valores de KPI (Clicks, CTR, Leads). Resuelven la dispersión integrando los resultados de forma foránea en la base de datos maestra.

[ ARCHITECTURE ] Forma de Implementación Técnica

A nivel de código fuente en Java Spring, la implementación escala mediante la segregación de componentes por el patrón arquitectónico MVC/N-Capas garantizando flujos aislados de I/O de la DB. Se exponen a continuación ejemplos de código real del proyecto:

Capa de Dominio (Modelos de Entidad)

Instrucciones de clase marcadas con @Entity. Aquí se codifica con exactitud el esquema de las tablas vistas en el diagrama relacional, mapeando variables orientadas a objetos como columnas de MySQL.

@Entity
public class Campania {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Integer idCampania;
    
    @Column(nullable = false)
    private String nombre;
    
    @ManyToOne
    @JoinColumn(name = "idCliente")
    private Cliente cliente;

    @OneToMany(mappedBy = "campania")
    @JsonIgnore
    private List<Tarea> tareas;
}

Capa de Servicios (Core Rules)

Cerebro transaccional con @Service. Efectúa verificaciones sistemáticas antes de la persistencia de datos.

@Service
public class ServicioCampania {
    @Autowired
    RepositorioCampania repositorioCampania;
    
    // Crear campaña delegando la persistencia
    public Campania crearCampania(Campania campania) {
        return repositorioCampania.save(campania);
    }
    
    // Actualizar con reglas de modificación
    public Campania actualizar(int id, Campania nueva) {
        Campania existente = buscarCampaniaPorID(id);
        
        existente.setNombre(nueva.getNombre());
        existente.setTipo(nueva.getTipo());
        existente.setFechaInicio(nueva.getFechaInicio());
        existente.setFechaFin(nueva.getFechaFin());

        return repositorioCampania.save(existente);
    }
}

Capa de Controladores (Routing Front-End)

Los @RestController abren los endpoints de la API REST que exponen el backend a un frontend o aplicación cliente (por ejemplo en React o Angular).

@RestController
@RequestMapping("/campañas")
public class ControladorCampania {
    @Autowired
    private ServicioCampania servicioCampania;
    
    // Endpoint para listar a través de peticiones HTTP
    @GetMapping("/lista")
    public List<Campania> listar() {
        return servicioCampania.listar();
    }

    // Retorna una campaña en JSON puro
    @GetMapping("/buscar/{id}")
    public Campania obtener(@PathVariable int id) {
        return servicioCampania.buscarCampaniaPorID(id);
    }
}

[ API_TESTING ] Pruebas con Postman

Guía de Ejecución de Peticiones

Dado que el núcleo de AFBD.PuntoJS es una API REST pura, la forma correcta de validar el funcionamiento del Guardado, Consulta y Eliminación es utilizando una colección de Postman para inyectar datos simulando las llamadas de un FronEnd:

>>> 1. Verificación del Host

Asegúrate de que el log de Spring Boot de tu IDE (IntelliJ o Eclipse) marque "Started Application...". Por defecto, tu base de datos y la API responderán en el localhost:

Base URL: http://localhost:8080 ó http://localhost:6969

>>> 2. Prueba de Creación (POST)

En Postman, crea un Request de tipo POST y elige en las pestañas del Body la opción raw formato JSON. Pega un Payload válido de la entidad a probar y dispara (Send).

// Ruta: POST http://localhost:6969/campañas/crear

{
    "nombre": "Campaña de Promoción SEO de Verano",
    "tipo": "Motores de Búsqueda (Google Ads)",
    "fechaInicio": "2026-06-15",
    "fechaFin": "2026-08-31"
}

>>> 3. Lectura y Comprobación (GET)

Para comprobar que Hibernate y MySQL persistieron los datos correctamente, crea en Postman una petición GET limpia y lánzala hacia tu lista. La consola te retornará un bloque JSON con IDs automáticos:

// Ruta: GET http://localhost:6969/campañas/lista
